Building a pfSense Firewall from Scratch
Steven Bush  ·  APR 2025  ·  8 min read

When I first started building my homelab, the cheapest consumer routers had minimal security controls and no real visibility into traffic. That changed when I moved to pfSense — and I've never looked back.

Why pfSense?

pfSense is a free, open-source firewall distribution based on FreeBSD. It gives you enterprise-grade features: stateful packet inspection, VLAN support, IDS/IPS integration, VPN, and a clean web interface — all on commodity hardware.

Hardware I used

I repurposed an old Dell Optiplex with a dual-NIC Intel card. Minimum spec: 2 NICs (WAN + LAN), 2GB RAM, 8GB storage. pfSense is lightweight.

VLAN Segmentation

One of the first things I configured was network segmentation. My lab runs four VLANs:

VLAN 10 — Trusted workstations
VLAN 20 — IoT devices (CCTV, smart devices)
VLAN 30 — DMZ / servers
VLAN 40 — Attack/pentest lab (isolated)

No inter-VLAN routing by default. Firewall rules explicitly whitelist only necessary traffic — VLAN 30 can talk to VLAN 10 on specific ports, nothing else.

Key Hardening Steps

After the base install: disable the webConfigurator on WAN, change the default admin password, enable HTTPS with a self-signed cert, configure Bogon blocking, and restrict SSH to LAN only.

For intrusion detection, I integrated Suricata with the ET Open ruleset. pfSense packages make this a 10-minute setup. Alerts pipe directly into my Graylog instance.

Result

Total build cost under $80. Full visibility into every packet crossing my network, VLAN isolation, VPN endpoint, and IDS — all running on a box that was heading for the bin. That's the homelab mindset.

Securing CCTV Systems Against Remote Exploits
Steven Bush  ·  MAR 2025  ·  6 min read

IP cameras are everywhere — homes, businesses, government buildings. They're also some of the most insecure devices on any network. Default credentials, unencrypted RTSP streams, and open management ports make them low-hanging fruit for attackers.

The Common Vulnerabilities

In my CCTV installations, I consistently find the same issues out of the box: admin/admin credentials, HTTP management interfaces exposed on port 80, RTSP streams on port 554 without authentication, and UPnP enabled by default — punching holes through your router automatically.

My Installation Checklist

1. Change default credentials immediately
2. Update firmware before connecting to network
3. Disable UPnP on both camera and router
4. Place cameras on isolated VLAN (no internet access)
5. Use a reverse proxy or VPN for remote access
6. Enable HTTPS on NVR management interface
7. Disable unused services (Telnet, FTP, ONVIF if unused)
8. Configure motion alerts to local storage, not cloud

Remote Access — Done Right

Many installers configure port forwarding directly to the camera or NVR. This exposes the management interface to the entire internet. Instead, I set up a WireGuard VPN on pfSense — remote access requires a valid VPN connection first. No ports forwarded, no attack surface.

For clients who need mobile access, I configure a VPN client on their phone. Takes 5 minutes to set up, eliminates the exposure entirely.

Network Isolation

Cameras live on VLAN 20 in my setups. They can reach the NVR for recording, but nothing else. They cannot reach the internet (preventing data exfiltration to manufacturer servers) and cannot reach trusted workstations. A compromised camera stays contained.

Homelab
My Homelab Defence Stack — 2025 Edition
Steven Bush  ·  FEB 2025  ·  10 min read

A homelab is the best way to develop real security skills. Reading about firewalls is useful; configuring one at 2am while breaking your own network is how you actually learn.

The Stack

Firewall:    pfSense (Dell Optiplex)
IDS/IPS:     Suricata + ET Open rules
SIEM:        Graylog (Ubuntu VM)
Honeypot:    Cowrie SSH honeypot
Vuln Scan:   OpenVAS (weekly scheduled)
Log Forward: Filebeat → Graylog
Alerting:    Graylog → Telegram Bot

Detection Pipeline

Suricata runs inline on the firewall, inspecting all traffic. Alerts are forwarded via syslog to Graylog. I built custom dashboards showing: top talkers, blocked IPs, geographic origin of connection attempts, and CVE hits.

The Honeypot

Cowrie is a medium-interaction SSH/Telnet honeypot. I expose it on a dedicated IP in a DMZ VLAN. It logs every command an attacker types, captures any tools they upload, and records session recordings. The data is fascinating — and useful for tuning Suricata rules.

Weekly Drills

Every Sunday I run a simulated attack from my isolated pentest VLAN (VLAN 40) and check whether Suricata detected it, how it appears in Graylog, and whether the alert fired in time. This keeps detection rules sharp and response procedures practiced.

The homelab isn't just a learning tool — it's where I test every configuration before deploying it for clients.

Kali Linux Toolkit: The Tools I Actually Use
Steven Bush  ·  JAN 2025  ·  7 min read

Kali ships with 600+ tools. Most pentesters use maybe 15 consistently. Here are the ones I reach for on every engagement — and why.

Reconnaissance

nmap is still the king of network scanning. My go-to command for a thorough scan:

nmap -sV -sC -O -p- --min-rate 5000 -oA scan_output <target>

gobuster for directory bruteforcing web targets. enum4linux for SMB enumeration on Windows environments — still invaluable for Active Directory recon.

Exploitation

metasploit for structured exploitation. I use it primarily for validated CVEs in known software versions — not spray-and-pray. hydra for credential attacks on SSH, RDP, and web login forms.

Post-Exploitation

mimikatz (via Meterpreter) for credential dumping on Windows. linpeas and winpeas for privilege escalation enumeration — these scripts are incredibly thorough.

Network Analysis

wireshark for packet captures. tcpdump when I need to capture on a remote host without a GUI. responder for LLMNR/NBT-NS poisoning in Active Directory environments — this catches credentials faster than almost anything else.

The Golden Rule

Know your tools well enough to use them without cheat sheets. Know the concepts well enough to improvise when the tools fail. The tool is not the skill — understanding what's happening at the packet level is.

Zero-Trust Remote Access with WireGuard + pfSense
Steven Bush  ·  DEC 2024  ·  8 min read

Traditional VPNs (OpenVPN, IPSec) work — but WireGuard is leaner, faster, and uses modern cryptography. Combined with pfSense and a zero-trust access model, it's the remote access setup I now deploy for all clients.

Why Zero-Trust?

Legacy VPNs grant network-wide access once authenticated. Zero-trust means: authenticated VPN connection grants access only to explicitly defined resources — nothing more. A compromised device can't pivot freely through the network.

pfSense WireGuard Setup

1. Install WireGuard package in pfSense
2. Generate server keypair
3. Create WireGuard interface (e.g. tun_wg0)
4. Assign static tunnel IPs (10.0.0.1/24)
5. Add firewall rules: allow VPN traffic to specific hosts/ports only
6. Generate client config with client keypair + server public key

Client Config Example

[Interface]
PrivateKey = <client_private_key>
Address = 10.0.0.2/32
DNS = 10.0.0.1

[Peer]
PublicKey = <server_public_key>
Endpoint = <your_public_IP>:51820
AllowedIPs = 10.0.0.0/24, 192.168.1.0/24
PersistentKeepalive = 25

Access Control Rules

On pfSense, I create alias groups for each resource. Client A (employee) → can reach file server on port 445 only. Client B (admin) → can reach management interfaces on ports 22, 443, 8080. No blanket "you're on the VPN, you see everything."

The result: fast handshake (<100ms), modern cryptography (ChaCha20, Curve25519), and granular access control. Clients notice it's faster than the OpenVPN it replaced. I notice fewer support calls about VPN disconnecting.

Hardening Windows Server 2022 — A Security Checklist
Steven Bush  ·  NOV 2024  ·  9 min read

A fresh Windows Server install is not a secure Windows Server. Here's the exact checklist I run through every time I deploy one — whether in a client's office in Nairobi or a homelab rack.

Immediately After Install

✓ Apply all Windows Updates
✓ Rename Administrator account
✓ Disable Guest account
✓ Set strong password policy via GPO (14+ chars, complexity)
✓ Enable Windows Firewall on all profiles
✓ Configure NTP to reliable source

Disable Legacy Protocols

SMBv1 is responsible for WannaCry and countless breaches. Disable it:

Set-SmbServerConfiguration -EnableSMB1Protocol $false

Also disable: NetBIOS over TCP/IP (NIC properties → Advanced → WINS), LLMNR (via GPO), and NTLM where Kerberos is available.

Active Directory Hardening

Implement the principle of least privilege — no user should be a Domain Admin unless absolutely necessary. Create tiered admin accounts: Tier 0 (DC admin), Tier 1 (server admin), Tier 2 (workstation admin). Each tier uses separate credentials.

Enable Protected Users security group for privileged accounts — prevents credential caching, forces Kerberos.

Auditing and Logging

Enable: Audit Account Logon Events (Success + Failure)
Enable: Audit Logon Events (Success + Failure)
Enable: Audit Object Access (Failure)
Enable: Audit Policy Change (Success + Failure)
Enable: Audit Privilege Use (Failure)
Forward: Security event logs to SIEM

Windows Defender ATP

On Server 2022, enable Defender ATP, configure cloud-delivered protection, and enable tamper protection. An attacker disabling your AV should trigger an alert — tamper protection ensures it can't be silently disabled via script.

This checklist takes about 2 hours to implement thoroughly. Skipping it takes 5 minutes and costs potentially everything.
